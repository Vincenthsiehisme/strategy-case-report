# Phase 4–7 執行規格

> 此檔案在進入 Phase 4 前讀取。

---

## Phase 4｜Analysis Blueprint（AI 自主執行）

> **分析模型參考**：選用模型時，讀取 `references/model-library.md` 取得模型庫、選擇決策樹與 HTML 圖表實作規格。

逐案完成內部規劃，不輸出給使用者確認。Blueprint 是執行前的思考框架，不是交付物。

**每案 Blueprint 包含四個部分：**

**① 頁面結構規劃**
依輸出媒介列出每案的頁面清單（標題 + 本案具體策略切角）：

```
頁面 1｜Case in one line
  → 本案切角：[品牌]如何用[具體動作]解決[具體問題]

頁面 2｜The real problem
  → 本案切角：[哪個結構性力量]讓[品牌]在[時間點]無法再照舊做

頁面 3｜The tension
  → 本案切角：[哪個習以為常的假設]在這個案例中被推翻了

頁面 4｜The strategic reframe
  → 本案切角：[舊說法]被[新說法]取代，差異在[具體的選擇]

頁面 5｜The system
  → 本案切角：[哪些接觸點]被串連，整體路徑邏輯是[什麼]

頁面 6｜Why it lands
  → 本案切角：[哪些已驗證事實]支撐，[哪些部分]只是判讀

頁面 7｜What to steal
  → 本案切角：[哪 2–3 個具體方法]可被借用，各自的前置條件是[什麼]
```

A4 版每案約 7 頁，Slide 版約 7–9 頁（部分頁面可合併）。

**② 分析模型配置**
每案最多 3 個模型，每個說明：
- 放在哪一頁
- 解決什麼分析問題（不是「這個模型叫什麼」，是「這個模型幫我說清楚什麼」）
- 如何呈現（流程圖 / 對比表 / 矩陣）

若拿掉任一模型，分析仍完整，那個模型就是多餘的——移除它。

**③ 核心分析問題（Four Questions）**
逐案預設四個問題的方向：
- Real problem：「為什麼這個問題在[時間點]無法再被忽視？」
- Tension：「哪個業界長期接受的假設，在這個案例中被揭穿？」
- Reframe：「品牌選擇了什麼不同的框架看待同一個問題？」
- System：「這個選擇如何被落地成一套可運作的接觸點系統？」

**④ 策略句草稿**
「[品牌] 面對 [真實問題]，選擇 [策略動作]，而不是 [原本的做法]。」
這一句是整個案例分析的核心主張，應在 Blueprint 階段就確認方向，Phase 5 展開時以此為骨幹。

完成後直接進 Phase 5。

## Phase 5｜First-Round Analysis（AI 自主執行）

> **寫作規則**：執行前確認已讀取 `references/writing-rules.md`，內含標題規則、寫作語氣、禁止模式、論證型配圖規則。

逐案展開完整分析，固定 6 個分析單元。每個分析單元的最低標準見下表——低於標準視為未完成，必須補足後才能進 Phase 5.5。

| 分析單元 | 輸出內容 | 最低標準 |
|---------|---------|---------|
| The real problem | 1 個核心問題定義 + 說明為何品牌不能照舊做 | 必須包含至少 2 個可被指名的結構性力量（不是「市場競爭激烈」這種泛稱）；必須說明為什麼這個問題在**現在這個時間點**無法再被忽視 |
| The tension | 1 個 insight 句 + 矛盾來源說明 | insight 句必須命名一個「習以為常的矛盾」——換入 2–3 個同類品牌名字後明顯不成立；必須說明這個矛盾存在多久、為什麼市場接受了它 |
| The strategic reframe | 舊說法 vs. 新說法 | 兩個說法必須是實質不同的框架，不是措辭不同的同一個框架；新說法必須說清楚「品牌主動放棄了什麼」 |
| The system | 關鍵接觸點列表 + 整體路徑邏輯 | 接觸點數量 ≥ 3；必須說明接觸點之間的邏輯關係（不是清單，是路徑） |
| Why it lands | 已驗證事實 + 判讀部分 | 事實、判讀、推論必須分開標注，不可混用；已驗證事實必須有來源標注；判讀部分必須標注「初步判讀」；若公開佐證不足，縮短此單元，不用推論填滿 |
| What to steal | 2–3 個可借用方法 | 每個方法必須包含：具體第一步（動詞開頭）、邊界條件（在什麼情況下不適用）、前置條件（需要什麼才能執行）。三項字數總量須符合 `output-slide.md` Steal card 字數上限；無法精簡時「前置條件」移入來源附錄，不得因此壓縮字體 |

**寫作規則：**

**標題規則：**
- 分析單元的頁面標題（如「The real problem」）是功能標籤，不是主張
- 每頁的核心主張放在 insight box / quote box 的文字內，必須是完整主張句，不是摘要句
- 主張句格式：「[品牌] 選擇了 [X]，而不是 [Y]——因為它判斷 [Z]。」

**寫作語氣：**
- 判斷優先於描述：「Trip.com 押注的是⋯⋯」比「Trip.com 做了⋯⋯」更好
- 命名矛盾：「這個行業長期把 A 和 B 視為對立，Club Med 花了 20 年證明這個對立是假的」
- 說清楚邊界：每個 insight 說清楚「在什麼條件下成立，在什麼條件下不成立」

**禁止的寫作模式：**
- 「因為 AI 的崛起」「在數位化浪潮下」——這些是背景描述，不是結構性力量
- 把品牌的自述當作已驗證事實（必須標注「品牌自述，未經獨立驗證」）
- 把「合理推測」直接寫成已驗證結論

完成後直接進 Phase 5.5。

## Phase 5.5｜深挖層（AI 自主執行，不外顯）

在第一輪分析完成後、防守型批判之前執行。

> **讀取完整規格**：執行前讀取 `references/deepdive-protocol.md`，內含三軸追問完整序列、八類結構性力量清單、各軸收斂機制與停止條件。

**三軸概覽：**
- **軸 1｜根源追問**（Real Problem）：三層追問找到可被指名的結構性力量 + 時間點觸發邏輯
- **軸 2｜矛盾命名測試**（Tension）：替換測試 + 反駁測試，確認 insight 不通用且有邊界
- **軸 3｜可執行性測試**（What to steal）：確認每個方法有具體第一步 + 邊界條件

停止條件：三軸都通過測試，或已達最大追問深度（每軸最多 3 層）。
結果直接覆寫 Phase 5 內容，不外顯。

**⚠️ 執行警示**：若三軸執行後什麼都沒有修改，這本身是警示訊號——應回頭確認測試是否真正執行（特別是軸 2 替換測試，AI 應真正換入品牌名驗證，不是宣稱執行了）。

完成後進 Phase 6。

## Phase 6｜防守型批判與修正（AI 自主執行，不外顯）

切換策略長視角，對每案執行防守型批判。Phase 5.5 負責挖深，Phase 6 負責確保沒有寫錯、沒有過度推論、沒有事實混淆。

> **讀取完整批判問題庫**：執行前讀取 `references/critique-scorecard.md`，內含 Full Mode 批判問題庫（事實層 / 推論層 / 相關性層）、Sprint Mode 精簡版、Scorecard 內部格式。

**執行模式規則：**
- 預設使用 Full Mode（全部問題庫）
- 僅在使用者明示要求精簡時使用 Sprint Mode（只執行事實層批判）
- Sprint Mode 不得主動選擇，也不得因任務複雜而自行降級

**執行邏輯（批判內化，不外顯）：**
- 發現問題 → 直接修正分析內容
- 無法修正（資料不足）→ 標注「初步判讀，缺乏公開數據支持」
- 批判執行摘要寫入最終報告來源附錄（發現了什麼問題、如何修正、或為何無法修正）
- 批判完成後不停止、不輸出 Scorecard，直接繼續

**Phase 6 完成後的流程分支：**
- 輸出形式為 A4 PDF、Slide PDF 或 HTML → 進 Phase 6.5
- 輸出形式為純文字稿 → 直接進 Phase 7

完成後直接進 Phase 6.5。

## Phase 6.5｜Visual Brief（AI 自主執行）

若輸出為純文字稿，跳過此 Phase。

> **視覺系統參考**：決策前讀取 `references/visual-ref.md`，內含色彩系統、版型錨點、設計調性選擇框架。

AI 依**輸出媒介 + 案例調性**自主決定視覺系統，不輸出 Visual Brief，不等確認：

1. 從分析內容萃取案例調性：產業類型、情緒基調（科技感 / 溫度感 / 策略嚴肅感）、讀者身份（內部提案 / 對外報告）、報告的核心主張
2. 依調性決定色彩系統：主色（品牌信任感）、強調色（動能 / 行動感）、中性色（輔助資訊）
3. **依輸出媒介決定版型邏輯**：
   - A4 印刷報告 → 讀取 `references/output-a4.md`
   - 16:9 Slide → 讀取 `references/output-slide.md`
   - **兩者設計語言不可混用**
4. 直接進 Phase 7

視覺系統決策邏輯記錄在來源附錄。
