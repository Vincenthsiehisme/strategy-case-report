# 純文字分析稿格式

> 此文件在 Phase 6 確認輸出為純文字稿（路徑 D）時讀取，由 SKILL.md Reference 讀取規則觸發。  
> 適用情境：使用者在 Intake 選擇「純文字分析稿」輸出，或在 Phase 1 觸發路徑 D 關鍵字（「純文字」「文字稿」「不要視覺」）。Full Mode 與 Sprint Mode 均適用。  
> 純文字稿不包含模型圖表、視覺設計，只輸出分析判斷內容。不需要 HTML artifact，不輸出截點 JSON，不轉 report-renderer。  
> **v2.6 更新**：Why it lands 改為三欄格式（confirmed / estimated / unknown）；What to steal 改為 scenarios 結構。

---

**The real problem：**
[直接資料支撐的問題定義，標注來源。若為代理推論，加注「初步判讀：」前綴。]

**The tension：**
[insight 句。格式：[習以為常的現象]，但實際上 [深層矛盾判斷]。]

**The strategic reframe：**

新框架：[品牌選擇的新說法]
換框依據：[一句話說明為什麼這樣換]

**What to steal：**
方法一：[動詞開頭的方法標題]
  邊界狀態：[rooted / estimated / unknown]
  情境 A（[條件描述]）：第一步是 [具體行動]
  情境 B（[條件描述]）：需要先 [前置條件]
  情境 C（[條件描述]）：此方法不成立，因為 [原因]

方法二：[動詞開頭的方法標題]
  邊界狀態：[rooted / estimated / unknown]
  情境 A（[條件描述]）：第一步是 [具體行動]

（選配）**The system：**
[若資料足夠，列出關鍵接觸點與各點角色，說明點與點之間的路徑邏輯，不是清單]

  - [有直接資料佐證的事實]（來源：[標注]）

## 注意事項

**三欄結構規則（v2.6）**
- `confirmed` 欄的每一條必須有來源標注。沒有來源支撐的陳述不能放在 confirmed，應降級為 estimated
- `estimated` 欄的每一條必須說清楚 proxy（使用了什麼代理變數）和 assumption（推論成立的前提條件）
- `unknown` 欄不是缺陷，是分析結果的一部分。每條必須說清楚需要什麼資料、以及補齊後對結論的影響。空白或「待補充」是無效的 unknown

**What to steal scenarios 規則**
- 每個方法至少要有一個情境描述，說明在什麼條件下成立、第一步是什麼
- 「第一步」必須是可操作的具體動作（動詞開頭），「研究市場」「評估可行性」不算具體第一步
- 邊界狀態 `unknown` 時，必須在情境描述中明確標注「邊界條件待驗證，以下情境為初步判斷」

**其他規則**
- The tension 欄必須是判斷，不是觀察。「消費者越來越重視健康」是觀察，不符合格式要求
- Case in one line 不先講背景、不先講得了什麼獎
- 知識邊界聲明不能省略——即使整份報告大部分是 confirmed，也必須誠實說出最關鍵的一個 unknown
- 來源清單必須包含所有在分析中引用的資料，代理變數來源（proxy）用類型標注區分
