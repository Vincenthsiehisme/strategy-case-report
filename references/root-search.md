# 扎根搜尋規格｜Root Search Protocol

> **讀取時機**：Phase 3 Shortlist 確認後，進入 Phase 3.5 執行前。  
> **用途**：定義扎根搜尋的執行邏輯、三層問題架構、兩階段搜尋策略、替換案例判斷標準。  
> **核心原則**：扎根搜尋不是「搜更多」，是「搜不同的東西」。找不到直接資料不是失敗，是訊號，改找代理變數。

---

## 定位與分工

| 搜尋類型 | 時機 | 目標問題 |
|---------|------|---------|
| 表層搜尋（Phase 2） | Shortlist 前 | 這個品牌做了什麼？有沒有足夠資料可拆？ |
| 扎根搜尋（Phase 3.5） | Shortlist 確認後 | 這個品牌為什麼不能不做這件事？ |

兩種搜尋回答的是完全不同的問題，不可互相替代。

---

## 三層問題架構

每個案例需要回答三層問題。每層有對應的直接資料類型和代理變數類型。

### 壓力層｜Why now

**核心問題**：為什麼這個問題在這個時間點無法再被忽視？如果不做這件事，品牌在 X 年後會面對什麼具體後果？

**直接資料類型**：
- 財務壓力訊號：毛利率、市占率、直客比例的變化數字，帶時間點
- 競爭事件：新進入者出現的時間點、平台規則改變、技術條件成熟
- 外部衝擊：政策生效日期、產業結構性事件、總體經濟衝擊

**代理變數類型（直接資料找不到時）**：
- 關鍵字搜尋趨勢（Google Trends）：品牌名 / 品類名的搜尋量變化
- 競品是否持續投放廣告：持續投放代表需求存在
- 官網是否新增方案與 FAQ：方案增加代表客戶壓力上升
- 社群上抱怨與需求的密度：客訴密度是壓力的間接證據
- 職缺是否持續擴編或縮減：人力部署方向反映戰略重心

**判斷標準**：找到的資料能不能回答「為什麼是現在」——有具體時間點和機制才算扎根，「市場競爭加劇」不算。

---

### 張力層｜The real tension

**核心問題**：這個矛盾在產業裡存在多久？為什麼大家習以為常、沒有人解決它？

**直接資料類型**：
- 產業慣例的文字證據：業界媒體、協會報告把某個做法寫成「標準做法」的描述
- 挑戰者出現的時間點：誰第一個做了不同的選擇、什麼時候、結果是什麼
- 失敗案例：嘗試過相同方向但失敗的品牌，以及失敗原因

**代理變數類型（直接資料找不到時）**：
- 同類問題在其他產業的處理方式：結構類比，不是直接佐證
- 產業論壇 / 從業者社群的討論密度：討論持續但沒有解方，代表矛盾真實存在
- 學術或顧問報告裡對這個問題的定義方式：看它被怎麼描述、何時開始被描述

**判斷標準**：找到的資料能不能說明「這個矛盾存在多久、為什麼市場接受了它」——只有成功品牌的說法，沒有產業背景的對照，不算扎根。

---

### 邊界層｜Steal conditions

**核心問題**：這個方法在什麼條件下不成立？什麼樣的品牌用了會失敗？

**直接資料類型**：
- 類似策略在其他品牌或產業的執行結果（成功和失敗都要找）
- 執行這個方法的組織前提：需要什麼能力、資源、結構
- 明確的反例：看起來相似但結果不同的案例，差異在哪裡

**代理變數類型（直接資料找不到時）**：
- 定價結構 / 方案設計：能反映執行這個策略需要的商業前提
- 供應鏈或履約限制：某些策略在特定供應鏈結構下根本無法執行
- 法規限制：某些市場的法規讓這個策略不可行

**判斷標準**：找到的資料能不能說出「這個方法在 X 條件下不成立」——沒有邊界的方法是假方法，不是扎根的。

---

## 兩階段執行邏輯

三層問題各自依以下順序執行，不並行。

```
階段一｜直接資料搜尋
  搜尋策略：品牌名 + 財報 / 法說會 / 危機報導 / 失敗案例
  搜到 → 填入 root_evidence 對應層的 confirmed 欄位
  沒搜到 → 進階段二，不視為失敗

階段二｜代理變數搜尋
  搜尋策略：依上方各層的代理變數類型選擇搜尋方向
  填入 root_evidence 對應層的 estimated 欄位
  必須附上推論邏輯：「這個代理變數能支撐這個判斷，因為 ___」
  以及成立假設：「這個推論在 ___ 條件下成立」
```

兩個階段都完成後，對每層輸出一個狀態標注：

| 狀態 | 定義 |
|------|------|
| `rooted` | 有直接資料支撐，可作為分析的事實錨點 |
| `estimated` | 只有代理變數，推論有邏輯但沒有直接佐證 |
| `unknown` | 兩個階段都找不到有意義的資料 |

## 替換案例判斷標準

三層問題跑完後，AI 對每案進行「扎根評估」：

**評估邏輯**：

```
壓力層 unknown + 張力層 unknown → 主動提案替換
  原因：問題定義和 insight 的基礎都不存在，
        分析只能靠語言深度填充，不是真正扎根

壓力層 unknown + 張力層 estimated → 降級繼續，標注風險
  說明：在報告中明確標注「壓力層缺乏直接佐證，以下分析基於代理推論」

壓力層 estimated + 張力層 rooted → 正常繼續
  說明：張力有根，壓力靠代理，整體仍有分析基礎

邊界層 unknown（不論其他兩層狀態）→ 不替換案例，但 What to steal 的
  使用條件必須標注「邊界條件待驗證」，不輸出假裝確定的方法論
```

**替換提案格式**：

> 「[案例名] 的壓力層和張力層都找不到足夠的扎根資料——直接資料和代理變數都無法支撐為什麼這個問題在現在這個時間點存在。繼續分析的風險是：Real problem 和 Tension 的深度會是語言層面的，不是資料層面的。
>
> 我的提案是替換這個案例。目前可考慮的替代方向是 [AI 提出 1–2 個具體替代方向]。
>
> 要替換，還是繼續但接受標注風險？」

## 扎根摘要（Phase 3.5 的輸出）

三層問題跑完後，AI 輸出扎根摘要，格式如下：

```
【扎根摘要｜[案例名]】

壓力層：[rooted / estimated / unknown]
  關鍵發現：[1–2 句，說明找到什麼或找不到什麼]

張力層：[rooted / estimated / unknown]
  關鍵發現：[1–2 句]

邊界層：[rooted / estimated / unknown]
  關鍵發現：[1–2 句]

整體評估：[這案例的分析基礎是否足夠 / 是否需要替換]
```

這個摘要是 Phase 4 Blueprint 的輸入，不輸出給使用者——AI 帶著這個摘要進入 Blueprint，決定每案的「已知 / 可估 / 待查」分布。

---

## 執行宣告格式

Phase 3.5 開始時，AI 輸出一句宣告：

> 「Shortlist 確認，開始扎根搜尋。預計執行 [N] 步驟：[案例 1] 三層 × [案例 2] 三層 × [案例 3] 三層，共 9 層搜尋，每層最多兩個階段。」

N = 案例數 × 3 層 × 最多 2 階段，通常 9–18 步。

執行過程不輸出中間結果，完成後輸出扎根摘要（AI 內部使用），然後直接進 Phase 4。

---

## 來源記錄規則

扎根搜尋找到的所有資料，依以下格式記錄在 `root_evidence` 裡，供截點 JSON 和來源附錄使用：

```json
{
  "layer": "pressure | tension | boundary",
  "fact": "具體事實陳述（資料裡說的，不是 AI 的推論）",
  "source_type": "direct_data | proxy | inference",
  "source_ref": "來源名稱或 URL",
  "assumption": "若為 proxy 或 inference，成立假設是什麼"
}
```

`source_type` 的三種狀態是強制標注，不允許模糊處理。
